# TxtGame 首页设计规格

## 1. 项目背景

TxtGame 是一个多剧本文字推理游戏合集，每个剧本是独立的 SugarCube/Twine HTML 文件，位于 `games/<codename>/` 下。目前仓库已有 `galley-villa`、`island-death`、`terminal-mystery` 三个剧本。项目缺少一个统一的入口页面，玩家需要手动进入各剧本目录才能找到可玩文件。

## 2. 设计目标

- 提供一个风格统一的项目首页，作为玩家进入各个子剧本的入口。
- 与现有 TypeHelp/SugarCube 引擎的视觉气质保持一致（暗色、终端感）。
- 支持未来新剧本的扩展：新增剧本时，首页应能低成本自动更新。
- 保持实现简单、鲁棒，不引入不必要的运行时依赖。
- 支持 GitHub Pages 静态部署。

## 3. 用户决策摘要

通过 brainstorming 视觉伴侣确认的关键决策：

| 维度 | 选择 | 说明 |
|------|------|------|
| 首页定位 | 混合型 | 顶部项目品牌介绍 + 主体剧本卡片列表 + 可直接启动游戏 |
| 视觉风格 | 终端黑客风 | 暗色背景、等宽字体、绿色/琥珀色强调色，延续 SugarCube 终端感 |
| 数据来源 | 构建时自动生成 | 通过 Node 脚本扫描 `games/` 目录，从 `README.md` 提取元数据生成 `games.json` |
| 运行方式 | 本地服务器 / GitHub Pages | `index.html` 通过 `fetch('./games.json')` 加载数据，需静态文件服务器 |
| 实现范围 | MVP | 首页 + 生成脚本 + 基础错误降级，不引入搜索/过滤/进度等高级功能 |

## 4. 页面结构与内容

### 4.1 页面区域

```
┌─────────────────────────────────────────────┐
│  Hero 区域                                    │
│  - 项目标题：TxtGame                           │
│  - 副标题：多剧本文字推理游戏合集               │
│  - 一句话简介                                 │
│  - 统计：当前收录 N 个剧本 · M 个可玩          │
├─────────────────────────────────────────────┤
│  剧本列表区域                                 │
│  - 区域标题：> 选择剧本                        │
│  - 卡片网格（1 列 ~ 3 列响应式）               │
│  - 每个卡片：标题、题材/基调、状态、简介       │
│  - 点击卡片跳转至对应 HTML 文件               │
├─────────────────────────────────────────────┤
│  页脚                                        │
│  - 简短操作提示：每个剧本独立运行              │
│  - 项目链接：GitHub / README                  │
└─────────────────────────────────────────────┘
```

### 4.2 剧本卡片信息

每个卡片至少显示：

- **标题**：剧本中文名（如“嘉利别墅”）
- **题材/基调**：一句话概括（如“1936 英国古堡 · meta 心理恐怖”）
- **状态**：完整可玩 / 设计完成 / 早期探索
- **简介**：2-3 行剧本简介
- **跳转链接**：`games/<codename>/<html-file>`

元数据来源约定：从各剧本 `README.md` 的 frontmatter 或固定段落提取。若提取失败，使用安全默认值（标题 = 目录名，状态 = "未知"）。

## 5. 技术方案

### 5.1 文件结构

```
TxtGame/
├── index.html                        # 首页入口
├── games.json                        # 剧本元数据（生成产物，入仓）
├── scripts/
│   └── generate-games-json.js        # 扫描 games/ 生成 games.json
├── games/
│   ├── galley-villa/
│   │   ├── README.md                 # 数据源
│   │   └── TypeHelp.html             # 可玩文件
│   ├── island-death/
│   │   ├── README.md
│   │   └── island-death.html
│   └── terminal-mystery/
│       ├── README.md
│       └── ...
└── package.json
```

### 5.2 首页实现

- 纯静态 HTML/CSS/JS，无框架依赖。
- CSS 使用原生变量定义终端风配色：
  - 背景色：`#0a0a0a` 或 `#111`
  - 主文字：`#e0e0e0`
  - 强调色：`#33ff33`（终端绿）
  - 次要强调：`#ffb000`（琥珀色，用于状态/警告）
- JS 逻辑：
  1. 页面加载后 `fetch('./games.json')`。
  2. 解析成功则渲染剧本卡片网格。
  3. 解析失败则在页面中显示静态备用列表 + 错误提示，确保玩家仍能找到已有剧本。
  4. 卡片点击跳转至对应 HTML 文件。

### 5.3 数据生成脚本

`scripts/generate-games-json.js`：

1. 扫描 `games/` 下的所有子目录。
2. 对每个目录：
   - 读取 `README.md`。
   - 提取标题（`# 标题`）、题材/基调（首段或固定格式）、状态、简介、可玩文件路径。
   - 若找不到 `README.md` 或可玩 HTML，标记状态为 "配置缺失" 并跳过跳转。
   - HTML 文件发现规则：优先读取 README 中 `**可玩文件**：[\`<file>\`](<file>)` 指定的文件；若未指定，则取目录下唯一的 `.html` 文件；若多个 HTML 文件存在，则取与目录名一致的 `<codename>.html`；否则标记为缺失。
3. 输出 `games.json`，包含剧本数组和生成时间戳。

提取规则示例：

```markdown
# 嘉利别墅 (Galley Villa)

> 1936 年英国古堡暴风雪山庄 + meta 心理恐怖
> **状态**：完整可玩（原版 TypeHelp 译版）
> **可玩文件**：[`TypeHelp.html`](TypeHelp.html)
```

生成字段：

```json
{
  "generatedAt": "2026-06-29T12:00:00.000Z",
  "games": [
    {
      "codename": "galley-villa",
      "title": "嘉利别墅",
      "subtitle": "Galley Villa",
      "genre": "1936 年英国古堡暴风雪山庄 + meta 心理恐怖",
      "status": "完整可玩",
      "description": "你是一名调查员，面前是一台从已故同僚处回收的旧计算机……",
      "htmlFile": "games/galley-villa/TypeHelp.html"
    }
  ]
}
```

### 5.4 鲁棒性设计

- **生成失败安全**：脚本遇到解析错误时记录警告，但不中断，继续处理其他剧本。
- **首页加载失败降级**：`fetch` 失败或 `games.json` 格式异常时，首页显示硬编码的备用列表（至少包含当前已知剧本）。
- **可玩文件不存在**：生成脚本检查 HTML 文件是否存在；不存在时状态显示为 "文件缺失"，卡片禁用跳转。
- **新剧本兼容**：新增剧本只需按现有模板写 `README.md`，无需修改首页代码。
- **不完整剧本处理**：如 `terminal-mystery` 无单一可玩 HTML，其 README 中不声明可玩文件，卡片状态显示为“早期探索 / 不可启动”，点击跳转至该剧本 README 而非游戏。

## 6. 扩展性考虑

- 未来若要支持搜索/过滤，可在 JS 中基于 `games.json` 的字段直接扩展，无需改动数据结构。
- 未来若要支持玩家进度/存档，需要后端或 localStorage；当前 MVP 不涉及，但首页卡片预留了状态字段扩展空间。
- 未来若要支持 file:// 协议直接打开，可增加一个构建步骤将 `games.json` 内容内联到 `index.html` 中，当前先以 fetch 方式满足 GitHub Pages 需求。

## 7. 测试策略

- **L1 静态检查**：确保 `index.html` 中引用的 `games.json` 路径正确，无死链接。
- **脚本测试**：运行 `npm run build:games` 后，验证 `games.json` 包含所有有效剧本。
- **E2E 冒烟**：用 Playwright 打开首页，验证所有可玩剧本卡片可点击并正确跳转。
- **降级测试**：临时移除 `games.json`，验证首页仍显示备用列表。

## 8. 开发/部署流程

1. 新增剧本：在 `games/<new-game>/` 下创建目录、README.md、可玩 HTML。
2. 生成元数据：`npm run build:games`
3. 本地预览：`npx serve .` 或 `python -m http.server`
4. 提交：`git add index.html games.json games/<new-game>/`
5. GitHub Pages：自动部署仓库根目录下的静态文件。
